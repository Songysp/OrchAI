# OrchAI: The Evolution of HiveMind

이 문서는 OrchAI 프로젝트가 탄생한 배경과 초기 설계의 한계를 극복하고, 현재의 CLI 기반 오케스트레이터로 진화하기까지의 기술적 의사결정 과정을 기록합니다.

---

## 🚀 1. 프로젝트의 목적 (The Purpose)
OrchAI는 **"복잡한 개발 태스크를 여러 AI 에이전트가 협력하여 해결하는 시스템"**을 목표로 시작되었습니다. 한 명의 AI가 아닌, 각 분야의 전문가(Planner, Worker, Reviewer)들이 팀을 이루어 작업할 때 결과물의 품질이 극대화된다는 믿음에서 출발했습니다.

## 🏗️ 2. 초기 설계: 서버형 아키텍처 (The Legacy Blueprint)
초기에는 누구나 어디서든 접속할 수 있는 **"SaaS형 AI 개발팀"**을 구상했습니다.
- **아키텍처:** FastAPI 기반의 중앙 서버, PostgreSQL(영속성), Redis(상태 관리).
- **인터페이스:** Slack 및 Discord 어댑터를 통한 메신저 기반 지시체계.
- **설계 의도:** 인프라를 중앙에서 관리하여 사용자가 설정을 최소화하도록 함.

## 🚧 3. 마주한 어려움들 (The Friction)
개발 단계가 진행되면서 다음과 같은 심각한 장벽들을 마주했습니다.
- **경험적 단절 (Context Switching):** 개발자가 코드를 작성하는 곳은 터미널/IDE인데, AI 팀의 피드백은 Slack으로 오는 "주의력 분산" 문제.
- **인프라의 역설 (Infra Burden):** "로컬 앱" 수준의 요구사항임에도 불구하고, 데이터베이스와 메시지 큐 등 무거운 서버 환경이 필수적으로 요구되는 비효율성.
- **네트워크 레이턴시:** AI 응답 시간 + 메신저 API 통신 시간이 더해져 "즉각적인 반응성" 확보 실패.

## 💡 4. 해결책: CLI-First 피벗 (The Pivot Decision)
2026년 3월 28일, 우리는 모든 서버 인프라를 폐기하고 **"로컬 터미널 기반의 오케스트레이션"**으로 전면 전환하기로 결정했습니다.
- **새로운 설계:** 사용자가 명령을 내리는 터미널(`orchai`)이 곧 서버이자 클라이언트가 되는 구조.
- **핵심 기술:** `Hybrid Adapter Pattern` (로컬 CLI 드라이버와 API 드라이버의 공존).

## 🎯 5. 결정의 근거 (The Rationale)
우리가 이 피벗을 결정한 기술적/전략적 근거는 다음과 같습니다.
1.  **즉각성 (Immediacy):** `claude --print`와 같은 로컬 CLI를 직접 부리는 것이 API를 통하는 것보다 저렴하고 빠름.
2.  **Zero-Infrastructure:** 별도의 DB 설치 없이 로컬 파일(JSON/SQLite)만으로 상태를 관리하여 사용자의 설치 허들을 극적으로 낮춤.
3.  **개발 친화적 (Dev-Native):** 개발자가 가장 오래 머무는 '터미널' 안에서 AI 팀이 바로 옆에 있는 것처럼 소통하는 것이 가장 생산적임을 확인.

---

이러한 고찰을 거쳐 탄생한 **HiveMind**는 이제 단순한 봇이 아니라, 개발자의 터미널 환경에 유기적으로 통합된 **"AI 개발 사령부"**로 작동합니다.
